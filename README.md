## 🧠 微前端架构决策与思考

### 为什么选择“微前端”而非“单体应用”？
当时业务线迅速扩张，我们面临：
1.  各子产品团队需要**独立开发和部署**。
2.  技术栈需要**渐进式升级**，不能一刀切重写。
3.  **老系统（Vue）与新系统（React）需要共存**。

基于此，我们评估了`qiankun`和`Module Federation`，最终选择`qiankun`，主要因为：
- **老项目兼容性更好**：对Vue 2.x项目侵入性低。
- **运维成本可控**：与公司现有的发布系统能快速集成。

### 遇到的挑战与解决方案：CSS隔离
子应用间的样式冲突是经典问题。我们未采用简单的`scoped`，而是设计了**两层方案**：
1.  **基础隔离**：通过`qiankun`的沙箱和命名前缀。
2.  **团队规范**：在《微前端开发公约》中强制约定，各子应用**必须使用CSS-in-JS（如styled-components）或CSS Modules**，从源头杜绝冲突。

**这一决策虽增加了初期学习成本，但彻底解决了样式污染，为长期维护打下基础。**
